# CMS / CRS 项目面试回答参考

本文基于当前仓库的真实代码结构整理，目标不是把项目包装得更夸张，而是帮助你在面试中把已经落地的能力说得更准确、更有层次。

适用范围：

- 营销 H5 可视化搭建中台
- 低代码 / 配置化平台相关追问
- Vue3 + TypeScript + Monorepo + 工程化 + 性能优化相关问答

---

## 1. 项目可以先怎么定性

一句话版本：

这是一个基于 Monorepo 的配置化页面搭建平台，拆分为 `cms` 编辑端和 `crs` 渲染端，围绕页面 Schema、组件物料和运行时渲染做了一套相对完整的双端协同方案。

稍展开版本：

- `apps/cms` 负责可视化编辑、物料拖拽、右侧配置、页面预览和 Schema 管理
- `apps/crs` 负责页面运行时渲染，既支持独立页面加载，也支持预览态接收编辑端同步的 Schema
- `packages/types`、`packages/utils`、`packages/ui` 承担协议、工具能力和共享组件抽象

这个定位是可以稳说的，因为仓库里确实有：

- `apps/cms`
- `apps/crs`
- `packages/types`
- `packages/utils`
- `packages/ui`

## 2. 代码里能稳稳支撑的核心主线

### 2.1 为什么拆成 CMS 和 CRS 两个应用

可以这样说：

“我当时的核心思路不是把编辑器和页面渲染混在一个应用里，而是把编辑态和运行态拆开。编辑器需要拖拽辅助、选中态、右侧配置和调试能力，运行态则更关心页面纯渲染、按需加载和线上稳定性。所以我在工程上拆成了 `cms` 和 `crs` 两个应用，再用共享包承接公共协议和组件能力。”

代码锚点：

- `apps/cms/package.json`
- `apps/crs/package.json`
- 根目录 `package.json` 中的 `dev:cms`、`dev:crs`

### 2.2 为什么用 Monorepo

可以这样说：

“这里用 Monorepo 不是为了追新，而是因为这个场景天然适合共享协议和组件。编辑端和渲染端都要消费同一套 Schema、组件定义和工具能力，如果分仓维护，协议演进和版本同步成本会很高。Monorepo 可以把共享层放进 `packages/*`，保证演进节奏一致。”

代码锚点：

- `pnpm-workspace.yaml`
- `turbo.json`
- `packages/types/src/schema.ts`
- `packages/utils/src/schema-adapter.ts`

### 2.3 为什么用 Schema 驱动，而不是直接堆组件

可以这样说：

“我更希望把页面描述抽象成稳定的数据协议，而不是把编辑器行为直接绑定到某个组件树实现上。这样做的好处是页面可以序列化存储、回放、迁移，也方便编辑端和渲染端消费同一份页面描述。”

更进一步：

“对这种平台来说，真正的核心不是拖拽本身，而是页面协议有没有抽象清楚。协议清楚了，编辑、预览、保存、发布、迁移这些链路才能统一。”

代码锚点：

- `packages/types/src/schema.ts`

### 2.4 为什么 Schema 从 V1 演进到 V2

可以这样说：

“我不希望历史数据兼容逻辑散落在业务组件里，所以把版本演进收敛到了适配层。V2 的重点是把页面结构从简单数组演进成 `componentMap + rootIds` 的扁平结构，查找、更新、迁移和扩展都更方便。”

更稳的表达：

“V1 更像简单页面描述，V2 更偏可扩展的数据协议，后续挂条件渲染、事件、局部状态都会更自然。”

代码锚点：

- `packages/types/src/schema.ts`
- `packages/utils/src/schema-adapter.ts`

### 2.5 编辑态和预览态怎么同步

可以这样说：

“预览这块我用了 iframe 隔离编辑态和渲染态，编辑端维护页面 Schema，预览端通过 `postMessage` 接收同步消息并渲染。这样做的好处是运行上下文更干净，不容易被编辑器样式和事件污染。”

代码锚点：

- `apps/cms/src/components/PreviewIframe.vue`
- `apps/crs/src/views/PagePreview.vue`

### 2.6 运行时渲染怎么做

可以这样说：

“渲染端不是把整个页面写死，而是基于 Schema 动态解析。根节点由 `rootIds` 驱动，节点渲染递归下发到 `RenderNode`，组件按类型异步加载，这样能把运行时消费控制在统一入口。”

代码锚点：

- `apps/crs/src/components/SchemaRenderer.vue`
- `apps/crs/src/components/RenderNode.vue`

### 2.7 撤销重做怎么做

可以这样说：

“编辑器高频操作很多，我没有手写一套 history 栈，而是基于 `useRefHistory` 管理页面快照。与此同时，配置修改不是每次都立刻落历史，而是做了防抖提交，避免快照膨胀和无意义的历史记录。”

代码锚点：

- `apps/cms/src/store/usePageStore.ts`

可以顺带补一句：

“我关注的不只是能不能撤销，而是撤销粒度和编辑器性能之间的平衡。”

### 2.8 表达式能力和安全控制怎么做

可以这样说：

“配置化平台里经常会有条件显示、简单计算这类诉求，但我不想把能力放大成任意脚本执行，所以表达式引擎只支持有限的变量访问和基础运算，同时屏蔽了 `eval`、`Function`、`constructor` 这类危险能力。”

代码锚点：

- `packages/utils/src/expression.ts`
- `packages/utils/__tests__/expression.test.ts`

### 2.9 拖拽和物料约束怎么做

可以这样说：

“物料不是想拖多少拖多少，我在物料分组里做了组件数量约束，比如公告、浮层、在线客服这类单例组件会限制最大添加次数。拖拽层负责校验是否允许放入，避免编辑态无效配置进入页面协议。”

代码锚点：

- `packages/ui/src/materials/definitions.ts`
- `apps/cms/src/views/Decorate/hooks/useDragDrop.ts`

## 3. 面试高频问题和推荐回答

下面这部分可以直接拿来口语化复述。

### 3.1 为什么拆成 `cms` 和 `crs` 两个应用，而不是一个项目里直接切页面

推荐回答：

“主要是考虑编辑态和运行态的职责差异。编辑端会带拖拽、选中、配置面板、调试辅助这些能力，运行端则更关注页面纯渲染和线上稳定性。如果混在同一个上下文里，样式污染、事件干扰和调试复杂度都会上来。拆成 `cms` 和 `crs` 之后，再通过共享包统一协议和组件，边界会清楚很多。”

如果继续追问：

“预览场景下我用 iframe 做了同步，真实页面场景下 `crs` 可以独立按页面 ID 拉取 Schema 渲染，这两个链路都在当前代码里有体现。”

### 3.2 为什么用 Monorepo，而不是两个项目分别维护

推荐回答：

“因为这个场景最大的共享对象不是业务代码，而是协议层和共享能力。编辑端和渲染端都依赖同一套 Schema、组件和工具函数，如果拆仓，协议升级和兼容会很痛。Monorepo 的价值在于把共享层统一放进 `packages/*`，工程边界更清楚，也更利于统一构建和版本演进。”

### 3.3 为什么要做 Schema V1 到 V2 的迁移

推荐回答：

“配置平台一定会面临协议演进，如果每次升级都要求业务数据重做，成本很高。所以我把兼容收敛到了适配层，内部统一消费 V2。这样历史页面可以继续使用，新增能力也可以逐步往新协议上挂，不会让兼容逻辑污染业务组件。”

### 3.4 `componentMap + rootIds` 相比数组或树结构有什么好处

推荐回答：

“我更偏向把它看成一层扁平化索引。对于编辑器来说，频繁操作往往是选中、更新某个节点、插入、删除和做历史回放。用 `componentMap + rootIds` 之后，节点查找和局部更新更直接，也更适合后续扩展事件、条件和局部状态。相比深层嵌套树，工程上更好维护。”

### 3.5 预览为什么用 iframe

推荐回答：

“为了隔离编辑环境和渲染环境。编辑器这边有很多辅助逻辑，不适合直接侵入页面运行态。iframe 可以让渲染上下文更干净，我只需要同步 Schema，不需要把编辑器状态一起带过去。”

### 3.6 编辑器的撤销重做是怎么设计的

推荐回答：

“我基于 `useRefHistory` 去做页面快照管理，重点不是功能本身，而是控制历史记录的颗粒度和内存开销。像右侧配置这种高频改动，不适合每次输入都生成一条历史，所以我做了防抖提交，确保历史记录有意义。”

### 3.7 运行时渲染怎么兼顾扩展性和性能

推荐回答：

“运行端这边不是把全部组件一次性打进去，而是按节点类型去异步解析组件。这样既能维持统一的渲染入口，也能把组件加载拆开。Schema 负责页面描述，`RenderNode` 负责递归渲染，异步组件负责按需加载，职责是分层的。”

### 3.8 表达式能力为什么不直接用 `eval`

推荐回答：

“平台类系统里我更倾向于能力可控，而不是表达式越强越好。`eval` 的风险太高，边界也太大，不利于维护。我这里支持的是有限变量、有限运算和安全校验，保证能满足配置化场景的主要诉求，同时可控。”

### 3.9 你在这个项目里对团队的横向贡献是什么

推荐回答：

“除了业务实现，我更关注的是把平台做成可持续协作的工程。这里我的横向贡献主要有三块：第一，统一了 Schema 和共享类型，减少了双端理解偏差；第二，把编辑端和渲染端的公共能力收敛到共享包，降低重复开发；第三，把预览同步、迁移适配和表达式能力放到统一入口，减少后续需求堆叠带来的维护成本。”

如果你想更偏“团队提效”：

“我做的不只是一个功能，而是一套接入方式和演进方式，后面别人接着扩时成本会更低。”

### 3.10 如果再做一版，你会继续优化什么

推荐回答：

“如果继续迭代，我会重点补三块：第一，补更完整的物料接入规范和脚手架，降低接入成本；第二，把表达式、事件和条件渲染能力抽象得更完整；第三，把测试覆盖进一步前移到协议和渲染链路，降低平台演进风险。”

## 4. 哪些话可以说，哪些话要收一点

### 可以稳说

- “这是一个 `cms + crs + packages/*` 的 Monorepo 双端架构”
- “编辑态和运行态做了分层，预览通过 iframe + postMessage 同步”
- “Schema 做了 V1 到 V2 的迁移适配”
- “运行时采用递归渲染 + 异步组件加载”
- “编辑器用了历史快照和防抖提交控制撤销重做粒度”
- “表达式能力做了安全限制，没有直接开放任意脚本执行”

### 建议收着说

- “彻底解耦”
- “完全低代码引擎”
- “根除所有跨 iframe 问题”
- “O(1) 解析”
- “性能提升非常大”

更稳的替代说法：

- 把“彻底解耦”改成“做了分层和边界收敛”
- 把“根除”改成“规避了主要冲突点”
- 把“O(1)”改成“降低了查找和更新复杂度”
- 把“性能提升非常大”改成“减少了无效更新和整体渲染开销”

## 6. 一段比较稳的项目口播模板

“这个项目本质上是一个基于 Monorepo 的配置化页面搭建平台，我把它拆成了 `cms` 编辑端和 `crs` 渲染端。编辑端主要负责物料拖拽、页面编排、配置和预览，渲染端负责基于 Schema 做运行时渲染。为了让双端协作成本更低，我把页面协议、共享类型和工具能力统一沉淀到了 `packages/*`。在实现上，我重点做了三件事：第一，设计并演进了页面 Schema，从简单结构升级到更适合扩展的 V2；第二，用 iframe + postMessage 处理编辑态和预览态同步；第三，在编辑器侧通过历史快照、防抖提交和物料约束控制高频操作下的可维护性和稳定性。” 

这段口播的优点是：

- 有架构视角
- 有平台思维
- 有工程化意识
- 不会说得过满

## 8. 最后给你的回答原则

回答这类项目时，尽量遵循这三个原则：

1. 先说“为什么这么设计”，再说“怎么实现”
2. 先说“边界和职责”，再说“某个细节技巧”
3. 先说“对团队和后续迭代的价值”，再说“我写了哪些代码”

如果你按这个顺序说，面试官会更容易把你看成“主导型前端”，而不是“只会写功能的人”。
