## 1. 项目定位

这个项目本质上是一个面向营销 H5 场景的可视化搭建平台。

我把它拆成了两个独立但协作的应用：

- `apps/cms` 负责编辑、配置和预览
- `apps/crs` 负责运行时渲染

这样拆分的核心目的，不是单纯为了目录清晰，而是为了把编辑态和运行态彻底分层。

编辑器需要拖拽、属性配置、复杂表单和后台交互能力；渲染端只关心如何根据页面协议高效还原页面内容。最终结果是，后台编辑能力不会污染移动端渲染环境，渲染端也不需要背负编辑器的重依赖和复杂状态。

## 2. 架构拆分

项目采用 Monorepo 组织方式，核心结构如下：

- `apps/cms`：可视化编辑器
- `apps/crs`：H5 渲染端
- `packages/ui`：共享业务物料组件
- `packages/types`：页面协议、消息类型和通用类型
- `packages/utils`：工具函数与 Schema 迁移适配
- `packages/hooks`：可复用组合式逻辑

这种拆分的价值在于：

- 页面协议、物料组件和基础能力可以在编辑端与渲染端之间共享
- 双端不需要各自维护一套业务组件和数据结构
- 后续新增物料或调整协议时，修改边界更清晰

## 3. 页面协议设计

这个项目不是把页面结构直接写死在 Vue 模板里，而是把页面抽象成一套可消费的页面 Schema。

当前仓库已经演进到 V2 结构，核心数据由两部分组成：

- `componentMap`：按组件 `id` 存储组件节点
- `rootIds`：定义页面顶层渲染顺序

相比直接维护一棵深层嵌套树，这种结构在低代码场景里更稳定，原因有三点：

- 组件查找和编辑可以直接按 `id` 命中
- 渲染顺序和节点内容可以拆开处理
- 更适合做协议迁移、历史快照和运行时消费

同时，项目保留了 V1 到 V2 的迁移适配能力。也就是说，旧页面数据不会因为协议升级直接失效，而是通过 `packages/utils` 中的迁移逻辑平滑接入新结构。

## 4. 渲染引擎设计

渲染端 `apps/crs` 不直接写死某个页面长什么样，而是通过运行时解析 Schema 来还原页面。

核心流程是：

1. `SchemaRenderer` 根据 `rootIds` 启动页面渲染
2. `RenderNode` 根据节点 `type` 选择实际组件
3. 渲染端从 `@cms/ui` 异步加载对应物料组件
4. 组件按节点结构递归组装成真实页面

这套设计的关键价值不是“动态渲染”四个字本身，而是把渲染器和业务物料解耦开了。

渲染器只负责按协议消费数据，不负责维护具体业务组件细节；物料层只负责扩展新的业务区块，不需要修改底层渲染机制。这样新增一个营销物料时，重点工作落在共享物料和配置层，而不是去改渲染引擎本身。

## 5. 编辑态与运行态隔离

为了让预览结果尽量接近真实运行态，项目没有把预览直接混在 CMS 页面内部渲染，而是让 CMS 通过 `iframe` 嵌入 CRS 预览页，再通过 `postMessage` 同步页面 Schema。

这里的职责边界很明确：

- CMS 维护页面状态和编辑行为
- CRS 接收最新 Schema 并负责渲染预览结果

这样做有几个直接好处：

- 后台管理系统的样式和交互不会直接污染移动端预览环境
- 预览链路更接近最终运行态
- 编辑器和渲染器的职责更清晰，问题定位也更直接

这也是项目里非常值得讲的一个点，因为它不是一个 UI 细节，而是一条完整的架构决策。

## 6. 状态管理策略

CMS 端的页面状态统一收敛在 Pinia store 中，页面协议、当前激活组件、拖拽状态和预览同步相关数据都走同一条状态链路。

为了支持撤销和重做，项目基于 `useRefHistory` 管理页面历史快照，而不是手写一套独立的历史栈。同时结合防抖提交，避免每次细粒度输入都产生一条无意义历史记录。

这套策略适合可视化搭建类场景，因为它平衡了三件事：

- 编辑操作频率很高
- 状态变更需要可回退
- 快照数量不能无限膨胀

## 7. 共享物料层

`packages/ui` 不是一个纯展示型组件库，而是偏业务语义的共享物料层。

当前已经沉淀了轮播、公告、商品、图文导航、富文本、辅助线、弹窗、浮层、在线客服等营销场景常见区块。编辑端负责这些物料的配置入口和属性面板，渲染端负责消费它们对应的最终协议。

这种设计的意义在于：

- 物料能力在双端保持一致
- 页面编辑结果和运行态渲染结果更容易做到同源
- 新增组件时，工作主要落在共享层而不是双端重复开发

## 8. UI 与样式治理

这个项目没有强行要求后台端和移动端只使用一种 UI 库，而是按场景分层：

- `apps/cms` 使用 `Element Plus` 处理后台复杂交互
- `apps/crs` 使用 `Vant` 处理移动端基础交互
- `packages/ui` 尽量保持对具体 UI 框架无感
- `Tailwind CSS` 主要承担布局、间距和原子化样式能力

为了避免多套 UI 能力叠加后样式边界失控，项目又补了一层治理规则：

- 统一 Token 来源
- 限制 `.el-*` 与 `.van-*` 深度覆盖
- 限制 `!important`
- 通过脚本和 `lint-staged` 增加门禁

这一层治理能力很适合在面试里讲，因为它体现的不是“页面能做出来”，而是“系统能长期维护”。

## 9. 这个项目最值得讲的三件事

### 9.1 Schema 可演进

我做的不是一份静态 JSON，而是一套可以支撑跨端共享、运行时渲染和协议迁移的页面结构协议。

### 9.2 编辑态与运行态分层

我没有把预览直接塞进编辑器内部，而是通过 `iframe + postMessage` 保持运行环境隔离，让预览更接近真实交付结果。

### 9.3 共享物料与工程边界

我把物料组件、类型协议、工具函数和应用层拆清楚了，让系统后续可以继续扩展，而不是停留在“能演示”的阶段。

## 10. 适合面试时的总结说法

> 这个项目不是单纯做了一个页面编辑器，而是搭了一套可演进的页面协议、双端分层架构和共享物料体系。编辑器、预览、运行时渲染和样式治理，都是围绕这条主线展开的。

## 11. 面试 QA：项目难点与亮点

<details>
<summary>Q1：这个项目里你觉得最核心的架构亮点是什么？</summary>

**答：**

我认为最核心的亮点是把编辑态和运行态彻底拆开了。`cms` 负责页面搭建、属性配置和预览控制，`crs` 负责消费页面 Schema 并执行运行时渲染。这样做的价值不只是目录拆分，而是把后台编辑器的复杂交互和移动端渲染环境隔离开了。编辑器需要 Element Plus、拖拽、配置面板和管理逻辑，渲染端只需要专注于页面还原和性能。这样预览结果会更接近真实运行态，后续扩展和定位问题也更容易。

</details>

<details>
<summary>Q2：为什么页面协议要设计成 <code>componentMap + rootIds</code>，而不是直接用一棵树？</summary>

**答：**

因为可视化搭建场景的核心操作不是“展示树”，而是“高频编辑”。如果直接维护一棵深层嵌套树，组件查找、编辑、排序、迁移都会越来越重。现在这个结构里，`componentMap` 负责按 `id` 存储节点，`rootIds` 负责页面顶层顺序，查找和修改都能直接命中节点，做历史快照和协议迁移也更稳定。简单说，树结构更适合表达层级，Map 结构更适合承接编辑器里的高频状态操作，我这里是把两者的职责拆开了。

</details>

<details>
<summary>Q3：预览为什么要用 <code>iframe + postMessage</code>，而不是直接在 CMS 里渲染？</summary>

**答：**

因为预览的目标不是“看起来像”，而是“运行环境尽量真实”。如果把 CRS 直接塞进 CMS 页面里渲染，编辑器的样式、全局状态和交互逻辑很容易污染移动端预览结果。用了 `iframe` 之后，CRS 运行在独立上下文里，CMS 只负责同步 Schema，CRS 只负责还原页面。这样后台端和移动端的样式边界更清晰，也更接近真实交付链路。`postMessage` 在这里承担的是跨上下文的数据同步职责，而不是业务逻辑本身。

</details>

## 12. 面试 QA：项目基础追问

<details>
<summary>Q1：为什么这个项目要用 Monorepo，而不是两个独立仓库？</summary>

**答：**

因为这个项目天然存在跨端共享能力。页面协议、共享物料、工具函数和 hooks 都会被 `cms` 和 `crs` 同时消费。如果拆成两个仓库，这些公共能力要么重复维护，要么需要额外发布同步，协作成本会很高。Monorepo 的价值在这里不是“流行”，而是让共享层真正可复用，双端可以围绕同一套协议和组件体系演进。

</details>

<details>
<summary>Q2：共享物料层在这个项目里承担什么角色？</summary>

**答：**

`packages/ui` 在这个项目里不是普通组件库，而是共享业务物料层。它沉淀的是轮播、公告、商品、图文导航、富文本、浮层这类营销场景组件。编辑端负责这些物料的配置入口和属性面板，渲染端负责消费它们的协议并输出最终页面。这样做的好处是，编辑态和运行态围绕的是同一套物料定义，而不是两边各写一套类似组件。

</details>

<details>
<summary>Q3：撤销和重做是怎么做的？为什么不用自己维护 history 栈？</summary>

**答：**

我这里是基于 `useRefHistory` 管理页面 Schema 快照，再结合防抖提交控制历史记录的颗粒度。原因是可视化搭建的配置修改非常高频，如果每次输入都立即入栈，历史记录会膨胀得很快，而且大部分记录都没有意义。相比手写一套 history 栈，我更关注的是“什么时候提交快照”和“什么变更值得进入历史”，这样更适合编辑器场景。

</details>

<details>
<summary>Q4：新增一个营销物料时，系统要改哪些地方？</summary>

**答：**

正常情况下，新增物料的主要工作落在共享层和配置层。先在 `packages/ui` 增加物料组件和导出，再在 CMS 侧补齐物料配置、默认值和属性面板，最后让 CRS 渲染端能够按 `type` 找到并消费这个组件。渲染引擎本身不应该因为新增一个业务物料就反复修改，否则说明系统边界没有拆干净。

</details>

<details>
<summary>Q5：你是怎么处理多套 UI 库并存的问题的？</summary>

**答：**

我的原则不是强行统一成一套 UI，而是明确分层。CMS 用 Element Plus 解决后台复杂交互，CRS 用 Vant 解决移动端基础交互，共享物料层尽量不直接依赖具体 UI 框架。Tailwind 则主要做布局和原子样式，而不是承担整个主题系统。除此之外，我又加了一层 UI 治理规则和脚本门禁，去限制深度覆盖、`!important` 和 token 漂移，避免多套 UI 长期叠加后把样式系统拖垮。

</details>

## 13. 七年前端面试视角：业务结果怎么讲

如果是七年前端面试，面试官通常不会满足于“你做了什么”，而是会继续追问“这个系统做成之后到底带来了什么变化”。

这里有一个原则：

- 有真实数据就讲真实数据
- 没有精确数据就讲相对变化
- 不要为了显得厉害去硬编指标

如果你手里没有特别完整的业务数据，建议按下面这个口径来讲。

### 13.1 效率提升

这类平台最直接的价值不是页面炫不炫，而是把活动页生产方式从“研发逐页实现”变成“运营先完成大部分搭建，研发只处理少量定制逻辑和接口对接”。

**答法示例：**

这个平台落地之后，页面生产链路最大的变化是研发不再从零手写每个活动页。很多标准化营销页面可以先由运营通过搭建平台完成主体结构和样式配置，研发更多是补接口、补个别定制逻辑和做发布兜底。对团队来说，本质上是把重复性的前端页面实现工作前移成了可配置能力。

### 13.2 复用提升

这个项目的第二个价值，是把“页面开发”变成了“物料复用”。

**答法示例：**

以前每做一个活动页，轮播、公告、商品区、图文导航这些模块虽然长得像，但实现常常是复制粘贴后再改。现在把这些通用区块沉淀进 `packages/ui` 之后，编辑端和渲染端围绕的是同一套物料协议，后续新增页面更多是在组合现有物料，而不是重复写页面。

### 13.3 稳定性提升

除了效率和复用，稳定性也是很重要的业务结果。

**答法示例：**

这个项目做完之后，编辑态和运行态分层带来的直接收益是联调偏差变少了。以前后台里看到的效果和移动端真实效果容易有样式偏差，现在预览链路就是独立的 CRS 渲染环境，问题更容易提前暴露，也更接近最终交付。

## 14. 七年前端面试视角：关键技术取舍怎么讲

七年前端面试里，面试官会特别在意你有没有做过“取舍”，而不是只会堆方案。

### 14.1 为什么不是一个应用里直接预览，而是 `iframe + postMessage`？

**答法示例：**

我当时优先考虑的是运行环境隔离，而不是开发时图省事。如果直接在 CMS 内部渲染 CRS 组件，短期看开发会简单一点，但长期会遇到样式污染、运行上下文不纯、编辑器逻辑侵入渲染器这些问题。`iframe + postMessage` 的代价是通信链路要自己维护，但它换来的是清晰的边界和更接近真实运行态的预览结果。对这种搭建平台来说，我认为这是更稳的取舍。

### 14.2 为什么协议要做成 `componentMap + rootIds`，而不是只保留树？

**答法示例：**

我不是为了结构新颖才这么做，而是因为编辑器场景和渲染器场景的关注点不一样。树结构表达层级很自然，但一旦进入高频编辑，查找、移动、替换节点会越来越重。`componentMap + rootIds` 的好处是把“存储节点”和“组织顺序”拆开了，编辑链路更高效，渲染层依然能还原出树形关系。它的代价是协议理解成本稍微高一点，但换来的是更好的可维护性和演进空间。

### 14.3 为什么没有一开始就上 SSR？

**答法示例：**

这个项目的核心问题不是内容 SEO，而是可配置页面的交付效率和运行时渲染能力。SSR 并不是不能做，但在这个阶段不是优先级最高的点。相比一开始就把复杂度引到服务端，我更关注的是先把页面协议、运行时渲染、预览隔离和共享物料体系立住。等这条主线稳定之后，再去看首屏优化、静态化或服务端预渲染，会更合理。

### 14.4 为什么后台和移动端允许多套 UI 库并存？

**答法示例：**

因为这两个场景的交互目标本来就不一样。CMS 是典型后台场景，表单、配置面板、复杂交互很多，用 Element Plus 更合适；CRS 是移动端渲染场景，用 Vant 会更贴近触屏交互。真正要管住的不是“库的数量”，而是边界和治理，所以我重点做的是共享层解耦、Token 统一和样式门禁，而不是机械式地强行统一框架。

## 15. 七年前端面试视角：风险与演进怎么讲

七年前端回答项目，不能只讲“已经做好了什么”，还要能讲“我知道哪里还不够好，以及为什么现在先这么做”。

### 15.1 协议演进风险

这个项目最大的长期风险之一，是页面协议一旦演进，历史数据就会成为包袱。所以我在设计上保留了迁移层，而不是假设 Schema 永远不变。这个思路本身就比“先不管，等出问题再说”更像成熟项目做法。

### 15.2 预览通信复杂度

`iframe + postMessage` 的边界很清晰，但通信链路本身也会带来复杂度，比如同步时机、消息定义和双端状态一致性。现阶段这套方案的价值在于稳定和隔离；如果未来预览交互继续变复杂，可以继续抽象消息层，或者评估更高阶的集成方式，但前提是不能破坏现有的职责边界。

### 15.3 编辑器复杂度持续增长

搭建平台类项目很容易出现“越做越像大泥球”的问题，所以我后面专门补了 UI 治理、共享层边界和脚本门禁。这类工作短期看不如新功能显眼，但它决定了系统能不能继续扩下去。

### 15.4 当前阶段的客观边界

如果在面试里被追问成熟度，可以这样说：

**答法示例：**

这个项目的主线已经比较清晰了，像双端分层、Schema V2、共享物料和 UI 治理都已经立起来了。但它并不是一个已经彻底封箱的系统，尤其编辑器侧还在持续做重构收口和协议升级后的边界清理。我对这件事的判断是，现阶段最重要的是先把主干架起来，再逐步把历史残留和细节债务压平。

## 17. 面试口述版

### 17.1 30 秒版

这个项目本质上是一个营销 H5 可视化搭建平台。我把它拆成了 CMS 编辑端和 CRS 渲染端，核心目标是把编辑态和运行态彻底分开。技术上我重点做了三件事：一是把页面抽象成可演进的 Schema V2；二是用 `iframe + postMessage` 做独立预览链路；三是把营销组件沉淀成共享物料层。它不是单纯做了个页面编辑器，而是在搭一套页面生产系统。

### 17.2 1 分钟版

这个项目是一个面向营销 H5 的可视化搭建平台，主要解决活动页重复开发、交付效率低和预览结果不稳定的问题。

我把系统拆成了两个应用：`apps/cms` 负责编辑、配置和预览控制，`apps/crs` 负责根据页面协议做运行时渲染。这样后台编辑器的复杂交互不会污染移动端渲染环境。

在设计上，我做了三件比较关键的事。第一，页面协议从树形结构演进成了 `componentMap + rootIds`，这样更适合高频编辑、历史快照和协议迁移。第二，预览没有直接塞进 CMS，而是通过 `iframe + postMessage` 让预览环境更接近真实运行态。第三，我把轮播、公告、商品区这些营销区块沉淀成共享物料层，让编辑端和渲染端围绕同一套协议工作。

所以这个项目最核心的价值，不只是把页面搭出来，而是把页面生产方式从“逐页研发实现”升级成了“共享物料 + 协议驱动 + 双端分层”的系统能力。

### 17.3 3 分钟版

这个项目的背景是，营销活动页需求频繁、样式变化快，如果每个页面都由前端从零实现，交付效率会很低，而且轮播、公告、商品区这类模块会不断重复开发。所以我做的不是一个单页面工具，而是一套面向营销 H5 的可视化搭建平台。

架构上，我把它拆成了两个应用。`apps/cms` 是编辑端，负责拖拽、配置、属性面板和预览控制；`apps/crs` 是渲染端，负责消费页面 Schema 并还原真实页面。这样拆的核心目的是把编辑态和运行态分层，避免后台系统的样式、状态和交互逻辑污染移动端页面。

页面协议上，我没有直接用一棵深层树来承载所有状态，而是演进到了 Schema V2，也就是 `componentMap + rootIds`。原因是编辑器场景里高频操作很多，比如选中、替换、移动、回退、协议迁移，如果只是深层树，查找和修改成本会越来越高。现在这个结构把节点存储和渲染顺序拆开了，更适合编辑器和运行时共同消费。

渲染层面，`crs` 不是写死页面模板，而是由 `SchemaRenderer` 和 `RenderNode` 按节点 `type` 动态选择共享物料组件，再递归组装页面。这样渲染器只负责协议消费，新增营销物料时主要改共享层和配置层，不需要反复侵入底层渲染引擎。

预览链路上，我没有为了省事把 CRS 直接嵌进 CMS，而是选择了 `iframe + postMessage`。它的成本是通信链路要自己维护，但好处是职责边界很清楚，预览环境更接近真实运行态，也更容易提前暴露样式和交互问题。

工程治理上，这个项目还处理了多套 UI 栈并存的问题。CMS 用 Element Plus，CRS 用 Vant，Tailwind 主要承担布局和原子样式。我没有强行把所有东西统一成一套，而是通过共享层解耦、Token 统一、限制 `.el-*` 和 `.van-*` 深度覆盖、限制 `!important`、再配合脚本门禁去管住样式边界。

所以如果要总结，这个项目真正的价值不是“做了一个拖拽页面”，而是搭了一套可演进的页面协议、双端分层架构、共享物料体系和持续治理机制。它解决的是页面生产方式的问题，而不只是某一个活动页的实现问题。

## 19. 面试 QA：按简历逐点追问怎么答

> 这一节专门对应简历里的“项目描述”和 6 条职责。面试官如果顺着你简历逐点展开，可以直接按下面口径回答。

<details>
<summary>Q1：你简历里写“基于 Monorepo 体系将编辑器与渲染引擎解耦”，具体在项目里是怎么做的？</summary>

**答：**

我不是只在目录上把项目拆开，而是把职责边界也拆开了。`apps/cms` 只负责编辑态能力，比如拖拽、属性配置、组件选中、历史记录和预览控制；`apps/crs` 只负责消费页面 Schema 做运行时渲染。公共能力放在 `packages` 下，例如 `ui`、`types`、`utils`、`hooks`。这样做之后，编辑器的 Element Plus、复杂管理逻辑和状态不会直接污染移动端渲染环境，渲染端也不需要背编辑器依赖。这个“解耦”本质上是编辑态和运行态的分层，而不是简单拆成两个目录。

</details>

<details>
<summary>Q2：你写“设计标准化页面 Schema/DSL 协议”，这里的协议到底长什么样？</summary>

**答：**

项目里页面不是直接写死在 Vue 模板里，而是抽象成一套页面协议。当前主结构是 `componentMap + rootIds`。`componentMap` 负责按组件 `id` 存储节点内容，`rootIds` 负责定义页面顶层渲染顺序。每个节点会描述自己的 `type`、`props`、样式配置和子节点关系。这样编辑器修改的是一份结构化数据，渲染端消费的也是同一份数据。严格说它更偏页面 Schema，而不是一门完整语言，所以我简历里写 Schema/DSL，是为了兼顾招聘语境和项目真实形态。

</details>

<details>
<summary>Q3：为什么协议不用一棵普通树，非要做成 <code>componentMap + rootIds</code>？</summary>

**答：**

因为这个场景的核心不是“展示层级”，而是“高频编辑”。如果只是一棵深层树，编辑器里做选中、查找、替换、移动、撤销重做时，节点定位和更新成本会越来越高。现在拆成 `componentMap + rootIds` 之后，节点内容和渲染顺序被分开了，按 `id` 查找和修改更直接，也更适合做历史快照和协议迁移。简单说，树适合表达层级，Map 更适合承接编辑器里的高频状态操作。

</details>

<details>
<summary>Q4：你简历里写“物料引擎”“配置驱动”“Headless 思路”，这一块在项目里具体指什么？</summary>

**答：**

这里的重点不是做一个通用 UI 组件库，而是沉淀营销场景下的共享物料，比如轮播、公告、商品区、图文导航、富文本、弹窗、浮层这些。所谓配置驱动，就是编辑端改的是组件配置和协议数据，渲染端按数据去还原物料，而不是双端各写一套相似页面。所谓 Headless 思路，我在面试里不会把它说得太虚，我会解释成“尽量把物料的配置模型、默认值、渲染消费和具体后台控件解开”，也就是让共享层更偏业务语义和协议消费，而不是死绑某一套具体表单或 UI 表现。

</details>

<details>
<summary>Q5：你写“基于 VueDraggablePlus 封装可视化画板，采用透明遮罩拦截 DOM 拖拽”，这个问题为什么会出现？你怎么解的？</summary>

**答：**

因为 CMS 里做的是编辑态，但预览是通过 `iframe` 嵌入 CRS 页面的。拖拽场景下，如果鼠标直接进入 `iframe`，事件链路会被切断，导致拖拽中的选中、放置、高亮反馈不稳定。我的处理思路不是去强行打通原生事件，而是在拖拽过程中加一层透明遮罩，把当前交互先留在编辑器上下文里处理，等放置位置和状态确认后，再同步 Schema 给 CRS。这样做的好处是交互链路更可控，也能减少跨上下文拖拽时的干扰问题。

</details>

<details>
<summary>Q6：你写“自研 CRS 渲染容器”，这里的渲染容器到底负责什么？</summary>

**答：**

CRS 的职责不是做页面编辑，而是做运行时消费。它拿到页面 Schema 后，会先由 `SchemaRenderer` 从 `rootIds` 开始组织页面，再由 `RenderNode` 根据节点 `type` 找到实际要渲染的物料组件，然后递归把整页组装出来。这里“渲染容器”的意思，不是安全沙箱那种概念，而是一个独立的运行时渲染环境。它的价值是把渲染器和业务物料解耦，新增营销物料时，主要改共享层和配置层，而不是反复侵入底层渲染逻辑。

</details>

<details>
<summary>Q7：你写“结合动态组件与异步分包，实现运行时解析与按需渲染”，能再具体一点吗？</summary>

**答：**

具体做法是，渲染端不会把所有物料在页面入口里全部写死，而是根据节点 `type` 去选择对应组件，并通过异步加载的方式把物料拉进来。这样首屏不需要把所有营销组件一次性打进去，运行时只消费当前页面实际需要的物料。这个设计和页面协议配合得比较紧，因为渲染器只关心“节点类型是什么、对应组件是谁、配置数据是什么”，而不需要感知业务物料的内部细节。

</details>

<details>
<summary>Q8：你简历里写“工具沉淀：统一调试、预览与构建流程”，这部分别讲虚的，具体做了什么？</summary>

**答：**

这块我会从“双端协作成本”来讲。因为项目是 Monorepo，如果没有公共工具链，编辑端、渲染端和共享包很容易各自跑偏。所以我会把类型、共享工具、组件导出、构建链路和调试方式尽量统一起来。比如公共类型和迁移逻辑放进共享包，预览链路固定走 CMS 到 CRS 的消息同步，构建上按应用和包的边界拆分，样式治理和门禁脚本也收进同一套工程规范里。它的目的不是显得“工具很多”，而是减少团队协作时的边界漂移。

</details>

<details>
<summary>Q9：如果面试官追问“你说显著缩短标准化活动页交付周期，这个价值是怎么来的”，你怎么答？</summary>

**答：**

我不会上来就报一个自己拿不准的数字。我会先讲链路变化：以前一个标准化活动页往往要前端从零搭页面、拼区块、调样式；做完这个平台后，很多通用页面可以由运营先完成主体搭建，研发更多只处理接口接入、少量定制逻辑和最终发布兜底。也就是说，效率提升不是来自“某个组件特别快”，而是来自页面生产方式变了，从逐页研发实现变成了配置化生产。

</details>

<details>
<summary>Q10：如果面试官质疑“这些是不是包装得太大，其实就是个页面编辑器”，你怎么接？</summary>

**答：**

我会承认它表层看起来像页面编辑器，但我会把重点拉回系统能力。真正有价值的不是拖拽本身，而是页面协议、编辑态和运行态分层、共享物料复用、预览隔离以及后续治理机制。这些能力一旦成立，服务的就不是某一个活动页，而是一类页面的生产方式。也就是说，编辑器只是表层交互，底下支撑的是协议驱动和双端协作体系。

</details>
